# B1 Evaluation Scripts

Пакет предназначен для автоматизированной проверки тренировочного задания B1 по топологии Published topology.

## Запуск

Локальная проверка на текущей ВМ:

```powershell
powershell -ExecutionPolicy Bypass -File .\local\b1-local-check.ps1 -ReportDir .\reports
```

Удалённая проверка с администратора/оценщика по списку хостов:

```powershell
powershell -ExecutionPolicy Bypass -File .\remote\b1-evaluate-remote.ps1 -ReportDir .\reports
```

## Что создаёт пакет

- b1-results.tsv — табличный отчёт по критериям.
- b1-detail.log — подробный журнал команд и вывода.
- b1-summary.txt — суммарная сводка по баллам.

## Предварительная настройка до начала выполнения задания

Перед началом выполнения задания должна быть выполнена базовая инфраструктурная подготовка. Иначе автоматическая удалённая проверка не сможет опросить целевые хосты.

### 1. Описание целевых хостов и их IP-адресов

Скрипт `B1\remote\b1-evaluate-remote.ps1` использует фиксированную карту хостов и IP-адресов.
Эти имена и адреса должны быть настроены заранее до начала задания.

- `sha-rtr01` — 10.21.20.1
- `sha-dc01` — 10.21.20.10
- `sha-fs01` — 10.21.20.20
- `sha-web01` — 10.21.10.11
- `sha-app01` — 10.21.10.12
- `sha-cl01` — 10.21.30.120
- `bj-rtr01` — 10.31.20.1
- `bj-dc02` — 10.31.20.10
- `bj-srv01` — 10.31.20.20
- `bj-cl01` — 10.31.30.120

### 2. Что нужно настроить на каждом хосте заранее

#### На всех целевых хостах

1. Присвоить хостам заданные имена (`hostname`) и IP-адреса.
2. Убедиться, что каждое имя разрешается либо через DNS, либо через `hosts`.
3. Включить PowerShell Remoting:

```powershell
Enable-PSRemoting -Force
```

4. Разрешить подключение по WinRM в брандмауэре:

```powershell
Set-NetFirewallRule -Name "WINRM-HTTP-In-TCP" -Enabled True
```

Если используется HTTPS, включите также `WINRM-HTTPS-In-TCP`.

#### На `sha-dc01`

- Должен быть настроен контроллер домена AD DS.
- Должны быть установлены роли DNS Server и AD DS.
- Должен быть готов домен `nb-b1.local` (или аналогичный домен, используемый задачей).
- Должны быть созданы DNS-зоны прямого и обратного разрешения.

#### На `bj-dc02`

- Должен быть второй контроллер домена AD DS и DNS для репликации.
- Должна быть настроена репликация с `sha-dc01`.

#### На `sha-fs01` и `bj-srv01`

- Должен быть установлен и настроен DHCP Server.
- Должны быть созданы DHCPv4-диапазоны и пропуск DHCP-ретрансляции, если используется несколько сегментов.
- Должны быть настроены файловые ресурсы SMB и NTFS-разрешения.

#### На `sha-cl01` и `bj-cl01`

- Должны быть клиентские машины, которые могут получить адреса из DHCP и подключиться к домену.
- На `sha-cl01` должен быть подготовлен финальный самопроверочный файл `C:\Skills\B1-selfcheck.txt` и/или Git-репозиторий, чтобы скрипт мог проверить финальную часть.

#### На `sha-web01` и `sha-app01`

- Эти хосты должны быть доступны по именам и IP, чтобы топология была полной.
- Если в задании предусмотрены веб/приложенческие сервисы, они должны быть развернуты.

### 3. Что нужно настроить в сетевой инфраструктуре заранее

1. Убедиться, что сегменты `10.21.x.x` и `10.31.x.x` связаны между собой так, чтобы машина-оценщик могла достучаться до всех целевых хостов.
2. Настроить маршрутизацию на `sha-rtr01` и `bj-rtr01`.
3. В контроллерах домена должны быть прописаны прямые и обратные DNS-записи для всех целевых хостов.
4. Адрес `198.18.100.10` (симулированный Интернет) должен быть доступен для проверки B1.1, если задача требует её симуляции.

### 4. Как проверить готовность до старта задания

#### 4.1. Проверка разрешения имен

С машины оценщика выполните:

```powershell
Test-Connection sha-dc01,sha-fs01,bj-dc02 -Count 1
```

#### 4.2. Проверка доступности WinRM

```powershell
Test-NetConnection -ComputerName sha-dc01 -Port 5985
Test-NetConnection -ComputerName bj-dc02 -Port 5985
```

#### 4.3. Проверка удалённого доступа PowerShell

```powershell
Invoke-Command -ComputerName sha-dc01 -ScriptBlock { hostname }
Invoke-Command -ComputerName sha-fs01 -ScriptBlock { Get-Service WinRM }
```

Если команды возвращают ответы, WinRM и удалённый доступ работают.

#### 4.4. Проверка ролей AD/DNS/DHCP

На `sha-dc01` и `bj-dc02`:

```powershell
Invoke-Command -ComputerName sha-dc01 -ScriptBlock { Get-ADDomain; Get-ADForest; Get-DnsServerZone }
```

На `sha-fs01` и `bj-srv01`:

```powershell
Invoke-Command -ComputerName sha-fs01 -ScriptBlock { Get-DhcpServerv4Scope; Get-SmbShare }
```

### 5. Где лучше запускать удалённую проверку

Рекомендуется запускать `B1\remote\b1-evaluate-remote.ps1` с отдельного административного хоста, который служит машиной оценщика.

- На этом хосте должны быть доступны все целевые машины по именам и IP-адресам.
- Лучше, если он находится в том же сегменте сети, где расположена контрольная инфраструктура AD/DNS/DHCP, то есть в сегменте `10.21.x.x` или `10.31.x.x`.
- Если есть возможность, разместите оценщика в сегменте `10.21.x.x`, потому что там находятся основная AD- и DNS-инфраструктура (`sha-dc01`, `sha-fs01`, `sha-rtr01`, `sha-cl01`).
- Желательно, чтобы он не был одним из проверяемых хостов, чтобы не смешивать роль оценщика и роль тестируемого узла.

#### Какая платформа лучше — Core или Windows Desktop?

- Рекомендуется использовать стандартную Windows Server машину или Windows Server Core, если у неё есть доступ к PowerShell и WinRM.
- Сервер Core подходит, если на нём настроен удалённый доступ и нет необходимости в графическом интерфейсе.
- Windows Desktop можно использовать, но лучше выбирать Windows Server, чтобы он был ближе по роли к инфраструктурной машине.
- Главное — чтобы на хосте оценщика был установлен PowerShell и он мог запускать `Invoke-Command` к целевым хостам.

Если отдельного администрационного хоста нет, запуск можно выполнить с `sha-dc01` только при условии, что с него доступны все остальные машины и настроены необходимые права.

### 6. Тестовый запуск после подготовки

После подготовки инфраструктуры попробуйте выполнить на машине оценщика:

```powershell
powershell -ExecutionPolicy Bypass -File .\remote\b1-evaluate-remote.ps1 -ReportDir .\reports
```

Если скрипт не может подключиться к какому-то хосту, он выдаст `WARN` и запись в `b1-detail.log`.

## Примечание

Часть критериев B1 носит ручной характер (например, LAPS, BitLocker recovery backup, drive mapping, права NTFS и проверка репликации). Для них скрипт выводит инструкцию на экран и помечает критерий как WARN, чтобы эксперт мог завершить проверку вручную.
