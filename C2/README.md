# C2 PNETLab Checker

Автоматизированная оценка Training C2 по 100 Measurement aspects (25 баллов).
Основной файл — `c2_check_ios.py`.

## Запуск

1. Запустите C2 lab в PNETLab. Имя running-сессии по умолчанию должно содержать `module-c`.
2. Проверьте параметры PNETLab и Cisco в `creds.json`.
3. Запустите из каталога `C2`:

```powershell
python .\c2_check_ios.py
```

Начать с критерия, aspect ID или порядкового номера:

```powershell
python .\c2_check_ios.py --start D
python .\c2_check_ios.py --start D10
python .\c2_check_ios.py --start 49
```

Если имя lab отличается:

```powershell
python .\c2_check_ios.py --lab c2
```

По умолчанию после завершения каждого аспекта (`A1`, `A2`, ... `G14`) скрипт
ждет нажатия `Enter`, прежде чем перейти к следующему аспекту. Для полностью
автоматического запуска без пауз:

Команда и ее отфильтрованный ответ выводятся сразу после выполнения. Поэтому
длительные DHCP- и failover-проверки отображаются по мере прохождения, а не
одним накопленным блоком в конце аспекта.

```powershell
python .\c2_check_ios.py --no-pause
```

E12 выполняет контролируемый WAN failover: временно отключает IR1 G0/3,
проверяет сохранение HQ route через IR2 и затем обязательно возвращает порт
командой `no shutdown`. Чтобы выполнить только недеструктивные проверки:

```powershell
python .\c2_check_ios.py --skip-disruptive
```

Перед оценкой F11 скрипт сам создает HQ traffic: DS1 отправляет ping к
198.51.100.100 с source 10.80.10.11. Сначала проверяется translation .81 на
IR1, затем при временно отключенном IR1 WAN создается новая сессия и
проверяется translation .82 на IR2. IR1 G0/3 восстанавливается в `finally`.
F11 оценивает факт создания translation с правильным inside global address;
результат ICMP показывается как диагностика, но не обнуляет корректный PAT.

F12 получает DHCP lease PC3/PC4, создает Internet traffic и проверяет PAT
inside local/global на BR1/BR2. F13 использует ISP1 как HTTP-клиент: через
telnet TCP/80 отправляет GET /healthz, ожидает OK-C2 и проверяет mapping на IR3.
F14 создает с PC1 межсайтовый трафик к PC3, PC4 и OPS-SRV, проверяет его
доступность и отсутствие NAT translations. Затем временно отключает IR1 G0/3,
до 60 секунд ожидает новую Internet PAT-сессию через IR2 с адресом
198.51.100.82 и обязательно восстанавливает интерфейс IR1 в `finally`.

Если аспект требует PC, SVR, JUDGE-SRV или ISP-side host, после результата
скрипт выводит краткий блок `Ручная проверка с хоста`: откуда запускать тест,
какие команды выполнить и какой результат ожидается. Такие подсказки также
показываются для частично автоматизированных проверок.

## Host-консоли

SVR1/SVR2 подключаются по их SSH console URL с учетной записью
`student` / `StudentPass`; значения находятся в `creds.json`. PC1-PC4
подключаются напрямую к VPCS telnet console URL. Для Linux SSH установите:

```powershell
python -m pip install -r .\requirements.txt
```

Автоматизированы VPCS DHCP-параметры и DNS (F5-F9), branch PAT traffic (F12),
guest/user connectivity (G3-G5), SSH/Telnet probes с JUDGE-SRV (A5, G1-G2),
SNMPv3 polling с SVR1/SVR2 (G10) и агрегированный host acceptance G13.
G13 запускается самостоятельно: четыре DHCP-проверки, полную DNS-матрицу,
разрешенную связность, guest isolation, SSH-open/Telnet-closed с JUDGE-SRV и
SNMPv3 authPriv с SVR1/SVR2. Результаты предыдущих аспектов ему не требуются.
Перед оценкой F5-F8 каждый VPCS выполняет `ip dhcp`, затем `show ip` опрашивается
каждые 3 секунды до 30 секунд. Проверка продолжается сразу после получения
ненулевого lease. Такой же механизм используется перед B20.

B20 автоматически обновляет DHCP lease на PC1-PC4, затем проверяет snooping
VLANs, разрешенные trust-порты, untrusted client ports и фактические bindings
на AS1-AS4. Ручное обновление DHCP для B20 не требуется.

C9/C10 автоматически проверяют HSRP tracking на DS1/VLAN10 и DS2/VLAN40:
один отказ должен оставить Active с priority 105, два отказа — переключить
роль при priority 90. Связность проверяется с PC1/PC2 до OPS-SRV; все временно
отключенные uplinks восстанавливаются в `finally`.
После изменения HSRP скрипт дает control plane 5 секунд на стабилизацию; VPCS
адрес PC считывается через `show ip`, затем SVR1 выполняет ping к PC. Ответ
клиента обязан пройти через новый HSRP gateway; это исключает ложный FAIL из-за
пустого интерактивного вывода VPCS ping.

B22 запускает 20 ping с PC1 к OPS-SRV параллельно с отключением DS1 G1/0.
Po12 должен остаться SU с G1/1(P), допускается не более двух потерь. После
теста G1/0 возвращается, и оба members должны снова стать (P).

C12 автоматически отключает tracked uplink DS3 G0/0, проверяет переход
DC VLAN130/150 на DS4 при priorities 90/100 и с JUDGE-SRV подтверждает ping и
SSH к удаленному management address. После теста DS3 восстанавливается и
должен вернуть Active role с priority 120.

D16 временно отключает IR1 G0/3, проверяет withdrawal BGP default и OSPF LSA
metric 10, наличие backup LSA IR2 metric 100 и HQ ping до Internet-test. В
`finally` WAN включается обратно и ожидается возврат primary BGP/OSPF default.

A3 выполняет реальный SSH login с SVR2 ко всем 15 Cisco как admin, проверяет
privilege 15, выполняет disable/enable с паролем и повторно проверяет privilege
15. Временный SSH_ASKPASS helper на SVR2 удаляется после каждого подключения.

A5 также проверяет все 15 устройств с SVR2: выполняется реальная SSH-
аутентификация admin и TCP probe порта 23. PASS требует успешный SSH и закрытый
Telnet; ручная проверка A5 не используется.

Для каждого аспекта скрипт сначала выводит ожидаемый результат и максимальный
балл, затем выполняет команды проверки и сразу печатает итоговый статус
`PASS`, `PART`, `FAIL` или `SKIP`. После вывода статуса следует пауза.

IOS-команды автоматически дополняются фильтрами `| include`, чтобы в evidence
оставались только строки, относящиеся к проверяемому параметру. Если аспект
проверяет несколько устройств, для каждого устройства выводится отдельный
блок: отфильтрованные команды и вывод, затем его статус `PASS`, `PART` или
`FAIL`. После этих блоков печатается общий результат аспекта.

Аспект B8 не требует заранее заданной карты портов: скрипт динамически находит
operational trunks командой `show interfaces trunk`, затем проверяет на каждом
`Administrative Mode: trunk` и `Negotiation of Trunking: Off`. Порты, для
которых образ IOS не выводит поле DTP, исключаются из пропорционального расчёта.

Для B16–B18 используется карта endpoint-портов опубликованной топологии:
AS1/AS2 G0/2, AS3/AS4 G0/1, DS3/DS4 G0/2. На них автоматически проверяются
PortFast/BPDU Guard, а на четырех PC-портах также port-security maximum 2,
secure MAC learning и violation mode restrict/shutdown.

Карта физических портов берется из опубликованной схемы PNETLab. В частности,
на IR2 G0/3 является WAN к ISP1, а G0/4 — transit к IR3. Для B19 скрипт
исключает все занятые по схеме HQ-порты, а оставшиеся физические интерфейсы
проверяет по `show interfaces status`: VLAN 998 и состояние `disabled`.

## Принципы оценки

- Используются operational и dedicated show-команды; `show running-config` не используется.
- `PASS` — выполнены все независимые проверки аспекта.
- `PART` — балл рассчитан пропорционально успешным проверкам.
- `FAIL` — проверка не пройдена.
- `SKIP` — аспект требует Linux PC/SVR, точной ручной привязки порта или контролируемого failover-теста экспертом.

Скрипт использует проверенные библиотеки подключения из соседнего каталога `C1` (`pnetlab_lib.py` и `checker_lib.py`), но не импортирует конфликтный `c1_check_ios.py`.
